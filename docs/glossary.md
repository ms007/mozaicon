# Glossary

Shared vocabulary for the Mozaicon project. Terms are grouped by topic. Data-model terms are also defined as Zod schemas in `src/types/`.

## Canvas

**Canvas** — The `<svg>` element rendered by `CanvasStage`, 1:1 with the icon coordinate system (its `viewBox` equals `documentAtom.viewBox`). Everything inside the SVG lives in viewBox coordinates; everything outside lives in app/screen coordinates. Boundary against _Artboard_ (the presentation wrapper that frames the Canvas) and _Document_ (the data model the Canvas projects). See [`canvas-chrome.md`](./canvas-chrome.md).

**Artboard** — The gesture-origin surface (`src/features/canvas/Artboard.tsx`). A `<div>` wrapper (`bg-card`, `rounded-xl`) that frames the Canvas and owns the tool lifecycle, pointer bridge, pointer capture, and the active tool's cursor class. A pointer press anywhere inside the Artboard — including the padding ring around the Canvas — begins a background gesture: _Drag-to-Draw_ when a draw tool is active, _Drag-to-Select_ when no tool is active. Coordinates map through the Canvas CTM and are used unclamped (out-of-viewBox values in the padding produce off-canvas geometry). The Artboard creates the canvas ref and renders `CanvasStage` as its child, threading the ref down as a prop. Interactive chrome mounted as a sibling of the `<svg>` must stop `pointerdown` propagation so it does not trigger an Artboard gesture. Boundary against _Canvas_ (the `<svg>` element it wraps) and _Page_ (the surrounding app chrome on `bg-muted`). See [`canvas-chrome.md`](./canvas-chrome.md).

**Pixel Grid** — The integer-position dot overlay rendered as the deepest layer inside the Canvas. Implemented as an O(1) SVG `<pattern>` + covering `<rect>`, viewBox-aware via a focused `selectAtom`. The grid is a spatial reference for pixel-perfect placement — it is not part of the document model and does not appear in export. Boundary against shape geometry (the grid is reference, not data) and _Artboard_ (the grid lives inside the `<svg>`, not in app-space chrome). See [`canvas-chrome.md`](./canvas-chrome.md).

## Rendering

**Element Tree** — The single document-to-SVG translation layer (`src/lib/svg/shapeElement.ts`). `documentElements(doc)` filters to visible shapes and maps each through `shapeToElement`, producing a `ShapeElement[]` — a discriminated union of `{ tag, attrs }` records with camelCase attribute keys combining geometry and paint. Adding a shape type means adding one case to `shapeToElement`; the three downstream printers consume the tree without per-shape changes:

- **Canvas ElementPrinter** (`src/features/canvas/renderers/ElementPrinter.tsx`) — maps a `ShapeElement` to JSX by spreading typed attributes onto the corresponding intrinsic (`<rect>`, `<path>`, …). If a new shape introduces a new SVG tag, the printer's switch needs an additional case and the `ShapeElement` union needs a new variant.
- **XML printer** (`src/features/export/serialize.ts`) — walks the element tree and emits raw SVG XML, kebab-casing attribute names (`strokeWidth` → `stroke-width`).
- **JSX printer** (`src/features/export/codegen.ts`) — sits after SVGO optimization; parses the optimized SVG string and reprints it as a typed React component with camelCase attributes. Because it consumes the SVGO output (not the element tree directly), adding a shape type doesn't touch this printer.

The public `shapeToElement` seam is the shape-specific boundary: everything upstream (schema, geometry) is per-shape; everything downstream (printers, pipeline, download) is generic. Export Parity is maintained by construction — canvas and export share the same element-choice logic (`chooseRectElement`) and paint logic (`shapePaintAttrs`).

## Interaction

**Gesture** — A transient, draft-sourced interaction on the canvas with a `null → populated → null` lifecycle. The draft atom is `null` at rest, populated while the gesture is in flight, and cleared back to `null` on commit or cancel. A gesture falls into one of two kinds based on input source: a _Pointer Gesture_ is driven by pointer-down-to-pointer-up (optionally promoted past a drag threshold) — _Drag-to-Draw_, _Drag-to-Select_ (marquee), _Drag-to-Move_, and Resize; a _Keyboard Gesture_ is driven by key-down-to-key-up (optionally across auto-repeat) — _Nudge_. Both kinds share the same contract: one draft atom, one _Gesture Adapter_ in the _Gesture Registry_, same `isAnyGestureActiveAtom` freeze. **Non-gestures:** tool-mode switches (e.g. choosing the rect tool), hover affordances (e.g. cursor changes, handle highlights), and overlays that contribute additional visuals during a gesture (snap guides, smart-distance indicators). These may read gesture state but do not own a draft atom and are not entries in the registry.

**Gesture Registry** — The ordered list of _Gesture Adapters_ in `src/store/atoms/gestures/registry.ts`. List position determines display-bbox precedence: during the `displayedSelectionBboxFromRegistryAtom` fold the first active adapter's `DisplayContribution` wins. Each adapter implements the `GestureAdapter` type (`name`, `draftAtom`, optional `displayBbox` callback) and contributes one of three tagged-union variants: `rect` (override the selection bbox with a concrete rect), `hide` (suppress the selection bbox entirely), or `passThrough` (defer to the next adapter in the list). Three derived atoms are computed from the registry: `isAnyGestureActiveAtom` (true iff any adapter's draft is non-null), `displayedSelectionBboxFromRegistryAtom` (the selection bbox to render, resolved by precedence), and `cancelGesturesAtom` (sets every adapter's draft to `null` in one write). The current precedence order is Marquee → Resize → Move → Nudge → Draw → PropertyStep. The registry is input-source-agnostic: it absorbs both pointer-driven adapters (Marquee, Resize, Move, Draw) and keyboard-driven adapters (Nudge). Adding a new gesture means writing an adapter and inserting it at the correct precedence position.

**Drag-to-Draw** — The gesture of creating a new shape on the canvas by pressing the pointer down, dragging, and releasing. Start and end point determine the shape's geometry (e.g. opposite corners of a rect). Distinct from _Drag-to-Move_ (translating an existing shape) and _Drag-to-Select_ (marquee selection).

**Drag-to-Select** — The marquee gesture started by a pointerdown on the canvas background while no draw tool is active. Crossing `DRAG_THRESHOLD_PX` (3 screen pixels) promotes the pending interaction into a marquee; sub-threshold pointerup is the _Click-Fallback_ instead. Hit-test is _Bbox_ intersection against the marquee rect, restricted to _Selectable Shapes_. Without Shift the gesture replaces the selection on `pointerup`; with Shift held at pointerdown it commits the symmetric difference against the _Base Selection_ snapshot (the modifier is captured once and stays fixed for the gesture's duration). Distinct from _Drag-to-Draw_ (creates a shape) and _Drag-to-Move_ (translates a shape).

**Layer Reorder** — The umbrella term for changing z-order via the Layers panel. Covers two sub-interactions: _Drag-to-Reorder_ (pointer: grab a layer row and drop it at a new position) and the four _Z-order shortcuts_ (keyboard: Cmd/Ctrl+] forward, Cmd/Ctrl+[ backward, Cmd/Ctrl+Alt+] to front, Cmd/Ctrl+Alt+[ to back). Both act on the current selection and commit as a single _History Entry_. Locked shapes are filtered out and act as fixed anchors; hidden shapes remain fully reorderable. Distinct from _Drag-to-Move_, which translates shapes on the canvas without changing z-order. Layer Reorder is a standalone panel interaction — it is not registered in the _Gesture Registry_ and does not trigger the active-gesture command freeze. **Panel-order ↔ array-index mapping:** top of the Layers panel = front = last index in `documentAtom.shapes`; bottom of the panel = back = index 0.

**Drag-to-Reorder** — The pointer sub-interaction of _Layer Reorder_. The user grabs a layer row and drops it at a new gap in the list. Dragging a selected layer moves the entire current selection as a contiguous block (preserving internal relative order); dragging an unselected layer first replaces the selection, then moves it alone. A non-contiguous multi-selection collapses into one block at the drop position. The document is not mutated during the drag — only a visual drop indicator and a drag overlay render. On drop, a single `moveShapeBlockCommand` commits the reorder. Implemented via dnd-kit (sensors, auto-scroll, `DragOverlay`); the list state stays in Jotai, not in dnd-kit's `arrayMove`. Distinct from _Z-order shortcuts_ (keyboard-driven, per-neighbor or block-insert semantics).

**Drag-to-Move** — The gesture of translating shapes by pressing the pointer down on any shape, dragging, and releasing. Pointerdown on a shape installs a pending interaction with a captured `startPoint`; crossing `DRAG_THRESHOLD_PX` (3 screen pixels) promotes it into a Drag-to-Move, sub-threshold pointerup is the _Click-Fallback_ instead. At promotion the _Move Set_ is captured as the snapshot of the current selection filtered to _Selectable Shapes_ (`visible && !locked`); pointerdown on a shape that is _not_ part of the current selection additionally dispatches `selectShapesCommand([id])` immediately before the first `moveDraftAtom` write, so the Move Set is the just-selected single shape — the selection-command must commit before `isGestureActiveAtom` flips to true, otherwise the gesture freeze swallows it. Shift held during the drag axis-locks the delta to the dominant axis since `startPoint` (read per frame). On `pointerup` a single `moveSelectionCommand` commits the final `{ ids, dx, dy }` — one _History Entry_. Escape and pointercancel clear the draft, leaving the document at its pre-gesture value. Distinct from _Drag-to-Draw_ (creates a shape) and _Drag-to-Select_ (changes the selection).

**Click-Fallback** — In a _Drag-to-Draw_ tool, the result of a pointer down/up that does not exceed the drag threshold (3 screen pixels). Inserts a default-size shape at the click position with the click point as the top-left corner, instead of a zero-size shape from the drag. For _Drag-to-Select_, the same sub-threshold pointerup dispatches `clearSelectionCommand` when Shift is not held, and is a no-op when Shift is held. For _Drag-to-Move_, the sub-threshold pointerup on a shape runs the click-to-select semantics — `selectShapesCommand([id])` without Shift, `toggleSelectionCommand(id)` with Shift — deferred from pointerdown so that an above-threshold drag on a multi-selection member doesn't collapse the selection before the move starts.

**Nudge** — The keyboard gesture of translating selected shapes by arrow keys. A key-down on an arrow key with a non-empty selection begins the gesture; subsequent arrow presses (including auto-repeat) accumulate into the running delta. The _Nudge Set_ is captured the same way as the _Move Set_: the current selection filtered to _Selectable Shapes_ (`visible && !locked`). On key-up (when all arrow keys are released) a single `moveSelectionCommand` commits the accumulated `{ ids, dx, dy }` — one _History Entry_ collapsing the entire arrow-key run. Escape clears the draft without committing. Suppressed when an editable element (input, textarea, contenteditable) has focus. Distinct from _Drag-to-Move_ (pointer-driven translation) and _Layer Reorder_ (z-order, not spatial position).

**Translation Gesture** — A gesture that translates selected shapes by a uniform `{ dx, dy }` delta, with a shared `null → populated → null` _Translation Draft_ lifecycle. The factory `createTranslationGesture(name)` in `src/store/atoms/gestures/createTranslationGesture.ts` produces the full surface for one translation gesture: a draft atom (`TranslationDraft | null`), a per-shape draft lookup (`draftForShapeAtom`, an `atomFamily` of `selectAtom` slices returning `{ dx, dy } | null` with structural equality on the slice), an `isActiveAtom`, and a _Gesture Adapter_ for the _Gesture Registry_ whose `displayBbox` translates the bbox of the draft's `ids` — the selection outline follows the shapes that actually move. Two adapters are currently instantiated: _Drag-to-Move_ (`src/store/atoms/gestures/move.ts`) and _Nudge_ (`src/store/atoms/gestures/nudge.ts`). To add a future translation gesture (e.g. paste-drag), call the factory with a new name, insert the returned adapter at the appropriate registry precedence position, and wire its `draftForShapeAtom` into `ShapeRenderer`, which reads each translation gesture's per-shape lookup explicitly and sums the offsets. Distinct from non-translational gestures (Resize, Draw, Marquee, PropertyStep) that produce per-shape geometry or selection, not a uniform delta.

**Translation Draft** — The shared transient type `{ ids: string[], dx: number, dy: number } | null` held by every _Translation Gesture_'s draft atom. `ShapeRenderer` applies the delta as a `<g transform="translate(dx,dy)">` wrapper for each shape in `ids`, giving live visual feedback without mutating the document. `ids` is the gesture's set (_Move Set_ / _Nudge Set_), captured at gesture start and intentionally frozen for the gesture's duration — the renderer and `displayBbox` key off `draft.ids`, never `selectedIdsAtom`. Selection changes from background events can't reach the gesture anyway while the command freeze is active, but the frozen set keeps that invariant explicit rather than incidental. On commit, the draft clears and the gesture dispatches one command — both current gestures commit via a single `moveSelectionCommand`. Defined as `TranslationDraft` in `src/store/atoms/gestures/createTranslationGesture.ts`. Replaces the former per-gesture draft types that move and nudge each defined independently.

**Nudge Draft** — See _Translation Draft_. The nudge adapter's draft atom is `nudgeDraftAtom`, produced by `createTranslationGesture('nudge')`.

**Gesture Sampler** — A per-gesture plain object created by `createGestureSampler(svg, scheduler)` in `src/lib/svg/gestureSampler.ts`. It caches the inverse screen-to-viewBox CTM once at construction (a single forced layout read via `svg.getScreenCTM()`) and exposes three methods: `toViewBox(screenPoint)` for synchronous, DOM-free coordinate conversion; `schedule(screenPoint, modifiers, callback)` to coalesce pointer samples to at most one delivery per animation frame, carrying the latest modifier state (shift/alt); and `stop()` to cancel any pending frame delivery. The scheduler is injectable (`FrameScheduler` interface — `request` / `cancel`) so unit tests can drive delivery synchronously without timers. A _Gesture Adapter_ creates a Gesture Sampler at gesture start and calls `stop()` on `pointerup` or cancel; between those events, every `pointermove` calls `schedule()` instead of writing the draft atom directly, ensuring draft writes are frame-throttled (≤1 per frame). The cached CTM is valid for the gesture's duration under current-viewBox semantics (no zoom/pan mid-gesture); when zoom/pan support lands, the single invalidation seam is the constructor — create a new sampler with a fresh CTM.

## Geometry

**Rect** — Axis-aligned rectangle `{ x, y, width, height }` in viewBox units. The geometric primitive shared by _Bbox_ results, _Drag-to-Draw_ output, and (structurally) the geometry portion of a `RectShape`. Defined in `src/lib/geometry/rect.ts`. Distinct from `RectShape`, which is a _Rect_ plus style and identity.

**Bbox** — The smallest _Rect_ enclosing a shape's rendered geometry. Computed by `bboxOf(shape)` in `src/lib/svg/bbox/`. Excludes stroke width and transforms — those are presentation concerns, not geometry. Group bbox is the union of child bboxes (planned).

**Selection bbox** — The union of the _Bbox_ of every currently selected shape, or `null` when the selection is empty. Computed by `bboxOfMany(shapes)`. Rendered by `SelectionOverlay` and is the anchor for selection-bound visuals (_Resize Handles_, dimension labels, snap guides).

**Resize Handle** — One of 8 visible circles rendered around the _Selection bbox_: 4 at corners, 4 at edge midpoints. Each handle has a transparent hit-area sibling with double the visual radius. Visual radius is `4 / viewBoxScale` so handles stay the same screen size at every zoom level. Rendered by the `ResizeHandles` component inside `SelectionOverlay`. No gesture wired yet — visual affordance only.

**Resize Anchor** — The point diametrically opposite the _Resize Handle_ being dragged. During a resize gesture the anchor stays fixed while the dragged handle and adjacent edges move to follow the pointer. For corner handles the anchor is the opposite corner; for edge handles it is the opposite edge midpoint.

**Resize Draft** — Transient per-shape geometry (`Record<id, Rect> | null`) held in `resizeDraftAtom` during a resize gesture. `ShapeRenderer` reads draft entries with fallback to the document, so the gesture renders live without committing to the document. On `pointerup` the draft clears and a single undoable `resizeShapeCommand` commits the final geometry.

**Move Draft** — See _Translation Draft_. The move adapter's draft atom is `moveDraftAtom`, produced by `createTranslationGesture('move')`.

**Move Set** — The subset of the current selection that actually translates during a _Drag-to-Move_ gesture: shapes in the selection that are also _Selectable Shapes_ (locked shapes filtered out). Captured at gesture promotion, frozen for the gesture's duration, stored as `moveDraftAtom.ids`. An empty Move Set blocks promotion — pointerdown on a lone locked shape, or on a non-shape target, cannot start a move.

## Corners

**Corner Style** — The interpolation mode for a rect's corners: `'rounded'` (circular arcs, the default) or `'smooth'` (superellipse Bézier curves). Stored in `corners.style` on each `RectShape`. The style drives the rect-vs-path routing in `chooseRectElement`: `'rounded'` with uniform radii produces a native `<rect rx>`, while `'smooth'` always produces a `<path>` with Bézier curves. When the style is `'rounded'`, the `smoothing` value is ignored regardless of its stored value. Schema: `CornerStyle` in `src/types/shapes.ts`. UI: a segmented control in the _Corners Section_, gated on non-zero radius. Distinct from `radii`, which controls the size of each corner, not its curve shape.

**Smoothing** — A 0–100 blending factor that controls how much a rounded corner deforms toward a superellipse (squircle) curve. `0` produces a standard circular arc; `100` produces a full superellipse quadrant. Only takes effect when the _Corner Style_ is `'smooth'` — the routing seam forces smoothing to 0 for `'rounded'` style. Stored in `corners.smoothing` on each `RectShape`. UI: a slider plus an editable percent field in the _Corners Section_, visible only when the corner style is `'smooth'`. Dispatches via `setSmoothingCommand`. Clamped to `[0, 100]` by the Zod schema and by the UI input.

**Squircle** — Informal name for a rectangle whose corners follow a superellipse curve rather than a circular arc. In Mozaicon, a squircle is a rect with _Corner Style_ `'smooth'` and _Smoothing_ > 0. The underlying math lives in `cornerPath` (`src/lib/geometry/corner-path.ts`), which blends between arc and Bézier control points based on the smoothing factor. Not a distinct shape type — it's a rendering mode of `rect`.

**Corners Section** — The properties-panel section (`src/features/properties/CornersSection.tsx`) that groups all corner controls for selected rects: the radius row (uniform or per-corner when expanded), the _Corner Style_ segmented control (gated on non-zero radius), and the _Smoothing_ slider (visible only in `'smooth'` mode). Renders nothing when no rects are selected. Multi-selection shows MIXED state when selected rects disagree on style or smoothing. Dispatches via `setCornerRadiusCommand`, `setCornerStyleCommand`, and `setSmoothingCommand`.

## Selection

**Selection** — The set of currently selected shapes, identified by IDs in `selectedIdsAtom`. Semantically a set: IDs are unique and ordered by document z-order (their position in `documentAtom.shapes`), not by click sequence. Session-local — not persisted by export or save. `selectedIdsAtom` is read-only; its module-private backing atom is accessible only through two write seams (`commitSelectionAtom` and `restoreSelectionAtom`), enforced structurally at compile time. All user-facing mutations go through _Selection-Commands_ and _Combined Commands_ so each effective change becomes one _History Entry_. Distinct from _Selection bbox_, which is the geometric envelope of the selected shapes.

**Selectable Shape** — A shape that qualifies as a candidate for user-driven selection: `shape.visible && !shape.locked`. Both _Drag-to-Select_ and click-to-select consult this predicate so the two paths can't diverge — hidden shapes don't render and shouldn't be ghost-selectable, locked shapes are explicitly protected from manipulation. Programmatic selection (e.g. an `addShapeCommand` selecting the freshly created shape) bypasses the predicate.

**Marquee Draft** — Transient marquee state held in `marqueeDraftAtom` during a _Drag-to-Select_ gesture: `{ pointerId, startScreen, startViewBox, current, additive, baseSelection? }`. A derived `previewSelectedIdsAtom` reads it to compute the hypothetical selection (replace with marquee hits when `additive` is false, symmetric difference against `baseSelection` when true). `SelectionOverlay` and _Resize Handles_ are suppressed while the draft is non-null, and `MarqueeOverlay` renders the rect. `selectedIdsAtom` stays untouched for the gesture's duration; on `pointerup` a single `selectShapesCommand` commits the final set — one _History Entry_. Escape and pointercancel clear the draft, leaving `selectedIdsAtom` at its pre-gesture value. Mirrors the _Resize Draft_ pattern; `marqueeDraftAtom` is registered in the _Gesture Registry_ via the marquee adapter, contributing to `isAnyGestureActiveAtom`.

**Selection-Command** — A command whose `apply()` returns a new `selection` but no `document`, e.g. `selectShapesCommand` (replace), `toggleSelectionCommand` (additive), `clearSelectionCommand`. Pushes a _History Entry_ in which the document axis is untouched (`before === after`). Distinct from a _Combined Command_ that touches both axes.

**Combined Command** — A command whose `apply()` returns both `document` and `selection` in one step, e.g. `addShapeCommand` (creates a shape and selects it). One _History Entry_, atomic for Undo/Redo — pressing Undo rolls the new shape out _and_ restores the prior selection together.

## Export

**Direct-Download Export** — Each export button is an immediate action: click = file download. There is no format toggle, preview dialog, or clipboard step. The user picks a target (SVG, TSX, 1x/2x/4x PNG) and the corresponding file downloads in one click. The dispatch seam is `performExport(doc, target)` (`src/features/export/performExport.ts`): both triggers — the export buttons (`src/features/export/ExportSection.tsx`) and the Mod+Shift+E shortcut binding — cross it, and it owns the target → pipeline → filename → download wiring. That seam is also the test surface for the wiring: `performExport.test.ts` verifies dispatch, filenames, scale mapping, and the error path; UI-level tests mock `performExport` and assert only the dispatch.

**PNG Scale** — A viewBox multiple used as the raster size for PNG export. Scale 1x renders the icon at its native viewBox pixel dimensions (e.g. 24×24); 2x doubles both axes (48×48); 4x quadruples them (96×96). Dimensions are rounded to the nearest integer pixel. The filename carries the scale suffix: `icon.png` for 1x, `icon@2x.png` for 2x, `icon@4x.png` for 4x. Implemented in `src/features/export/rasterize.ts`.

**Sticky Export Target** — The last-triggered export variant (`svg`, `tsx`, `png1x`, `png2x`, or `png4x`), persisted in `exportTargetAtom` for the duration of the session. The sticky target receives the `primary` button variant so the user can see which format they last exported. The keyboard shortcut (Mod+Shift+E) re-triggers the sticky target without clicking. Defaults to `svg`. Defined in `src/store/atoms/export.ts`.

**Export Parity** — The invariant that exported output renders identically to the canvas content, minus editor chrome (pixel grid, selection overlay, resize handles). Editor metadata (selection state, locked/visible flags, shape names) never appears in output. Hidden shapes are excluded; visible shapes serialize with their exact fill, stroke, and geometry. The canvas renderers and the export printers share the element-choice logic (`src/lib/svg/rectElement.ts`) and the paint logic (`shapePaintAttrs` in `src/lib/svg/shapeElement.ts`), so canvas and export agree on geometry and paint by construction. SVG and TSX are additionally structurally identical: the TSX component is generated by parsing the optimized SVG, so both formats carry the same tags, attributes, and values.

## Fields

**NumberInput** — A labeled numeric property field component (`src/components/NumberInput.tsx`). Renders a container-owned bordered control (28px height, elevated input surface, `focus-within` ring) around a borderless decimal-mode text input with optional leading/trailing affix slots. Uses a draft-then-commit model: typing and arrow presses mutate a local buffer; a single `onCommit(n)` fires on Enter and on blur (only when the value actually changed from what the buffer was anchored to, so an external update mid-edit isn't re-committed as the user's edit), collapsing edits into one downstream history entry. The optional `onChange(n)` emits the clamped, committable value on every keystroke and arrow step for live preview, and is restored to `value` on Escape, which discards the buffer. Arrow keys step by `step` (default 1), Shift ×10 for coarse, Alt for `fineStep` (defaults to the smallest increment the `precision` can represent, else 0.1 — so an integer field's Alt step is the base step, not a fractional no-op); fine wins when both modifiers are held. All number math delegates to `src/lib/util/number.ts`. Value-neutral — knows nothing about shapes or commands.

## History

**History Entry** — A record on the undo/redo stack capturing one command's effect. Fields: `label` (debug-only English string), `before` and `after` `Document` snapshots, plus `selectionBefore` and `selectionAfter` (`string[]`). Each entry covers both axes: a _Selection-Command_ leaves `before === after`, a document-only command leaves `selectionBefore === selectionAfter`, a _Combined Command_ moves both. Selection rides on the same stack as document mutations because each effective selection change is itself a user-visible Undo step (Figma-style). Defined in `src/store/atoms/history.ts`.

**Active Gesture** — A _Gesture_ whose draft atom is non-null, signalling that a pointer-driven interaction is in progress. Surfaced via `isAnyGestureActiveAtom`, which is derived from the _Gesture Registry_ — true iff any registered adapter's draft is non-null. While an _Active Gesture_ runs, _all_ commands no-op — document, selection, and combined alike — and `undoCommand`/`redoCommand` are likewise frozen. The freeze lives in `createCommand` itself, not only in undo/redo, so background events like a layers-panel click or `Cmd+A` can't yank selection out from under the in-flight gesture. `canUndoAtom` / `canRedoAtom` report false during the gesture; the toolbar's undo/redo buttons grey out. The pending change isn't a `HistoryEntry` yet — it commits on `pointerup`. To cancel a gesture, use Escape (separate mechanism). The invariant "Active Gesture iff draft atom is non-null" is verified by the at-most-one-active contract test in `registry.test.ts`.
